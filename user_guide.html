1   What This App Does

The HIOKI Data Logger connects to a HIOKI resistance meter (RM3544-01, RM3545, RM3542, DM7276, DM7275, IM7580A, etc.) through a USB or RS-232 serial cable and continuously reads resistance measurements. Every stable reading is:

  • Displayed live on-screen
  • Compared against your configured upper / lower limits (PASS or FAIL)
  • Accompanied by a Thai-language audio alert
  • Saved to a daily CSV file on this PC
  • Uploaded to the factory SQL Server database automatically

No manual intervention is required — the app auto-detects the meter, auto-reconnects after cable problems, and queues failed uploads for retry. It can also be deployed as a systemd service on Raspberry Pi for always-on operation.

3   Starting the App

  1. Plug the HIOKI meter into the PC with the USB / RS-232 cable and power it on.
  2. Open a terminal (or double-click the shortcut) and run: 
    python main.py
  3. The application opens full-screen and immediately begins scanning COM ports for a HIOKI device.
  4. Once found, the status bar turns green and measurements start automatically.
Note: Connect the meter before starting the app. If you plug it in after, the app will find it automatically within a few seconds.

6   Setting the Product Model

The app tags every measurement with a product model name before saving it. There are three ways to set the model:

A Barcode Scanner (recommended)

Point a USB barcode scanner at the product label. The model is decoded instantly and shown on the Model button. No clicks needed.

Format: The scanner must output Code 39 Extended with AIM /X escape sequences. The app strips the leading check digit and converts escape codes automatically.
Example: ?H60AGV/HFCWR/I-MM-4FHX-B5/D... → H60AGV(FCWR)-MM-4FHX-B5

B Model Button (manual)

  1. Click the Model button (top-left of the screen).
  2. Type or paste the model name into the dialog.
  3. Click OK.

If you paste a raw barcode string containing $ separators (e.g. FOD123$1SRG14R(BRK)-MM$15) the app automatically extracts the text between the first and second $.

C Auto-Prompt

If the model is empty when a stable reading is recorded, the app displays a model entry dialog automatically so no reading is lost without a label.

Barcode Decode Priority

All three model-entry paths (scanner, Model button, auto-prompt) run through the same unified decode function in the priority order below:

PriorityFormatDetectionExample inputResult
1 Dollar-delimited Contains $ FOD123$1SRG14R(BRK)-MM$15 SRG14R(BRK)-MM
2 AIM Code 39 Extended Contains /[A-Z] escape pair ?H60AGV/HFCWR/I-MM-4FHX-B5/D... H60AGV(FCWR)-MM-4FHX-B5
3 Plain text No $ or /X patterns SRG14R-MM-4FHX-B5 SRG14R-MM-4FHX-B5

AIM Code 39 Extended details: strips the leading check-digit character, converts /H→(, /I→), /L→/, etc., and stops at /D (field separator).

Persistence: The last used model is saved to gui_mode5_config.json and restored automatically on the next startup. If that file is missing, the app recovers the model from model_changes.csv.

7   Configuring Pass / Fail Limits

  1. Find the Lower Limit and Upper Limit spin-boxes on the screen.
  2. Click the value you want to change and type the new number, or use the up/down arrows. Values are in ohms (Ω). Range: 0.01 – 9999.00 Ω, resolution 0.001 Ω.
  3. The new limits are saved immediately to gui_mode5_config.json under the current model name. Switching to a different model will load that model's own limits automatically.
ScenarioJudgement
Reading < Lower LimitFAIL: Low
Lower Limit ≤ Reading ≤ Upper LimitPASS
Reading > Upper LimitFAIL: High
Non-numeric readingN/A
Warning: Make sure the Lower Limit is smaller than the Upper Limit. The app does not enforce this automatically.

8   Understanding Measurement Results

Stability detection

The app reads the meter every 500 ms. A reading is only recorded (logged + uploaded) when the same value is seen twice in a row — this confirms the meter's Auto Hold has locked on to a stable measurement. Unstable or transitional readings are silently discarded.

11   Connection Loss & Reconnection

The app handles cable disconnections and serial errors automatically:

MechanismDetail
Exponential back-off Retry delay starts at 1 s, grows by ×1.5 each attempt, capped at 60 s
Heartbeat check Sends *IDN? to the meter every 30 s to detect silent failures
Timeout limit 3 consecutive FETC? timeouts trigger a full reconnect cycle
Poll thread crash guard Any unexpected serial error causes a clean reconnect rather than a crash
Watchdog (Linux) Kills the app process after 30 s in uninterruptible (D) sleep state

While reconnecting the status bar shows Scanning ports (retry in Xs)…. No action is needed — plug the cable back in and the app finds it automatically.

12   Raspberry Pi Deployment

The app runs as a pair of systemd services on Raspberry Pi for always-on, auto-start operation.

Initial setup

  1. Copy the project to /home/pi/Hioki_data_logger/.
  2. Make the setup script executable and run it once as root: 
    chmod +x setup_services.sh
sudo ./setup_services.sh

The setup script automatically:

  • Installs system packages via apt (serial, ODBC libs, XCB cursor, GStreamer for MP3 audio)
  • Creates a .venv virtual environment with --system-site-packages
  • Installs all Python packages from requirements.txt
  • Verifies every module imports without error — exits with a clear failure message if any import fails
  • Writes and enables two systemd services
  • Starts both services and shows their status
GStreamer required for audio: MP3 playback via QMediaPlayer needs GStreamer on Linux. The setup script installs the required packages (gstreamer1.0-plugins-good, gstreamer1.0-alsa, gstreamer1.0-libav, etc.) automatically.

Services

ServiceRole
hioki-app.service Runs main.py, waits for X11 display, auto-restarts on crash
hioki-watchdog.service Kills the app if it enters uninterruptible (D) sleep for 30 s

Useful commands

sudo systemctl status  hioki-app.service
sudo systemctl restart hioki-app.service
sudo systemctl stop    hioki-app.service
journalctl -u hioki-app.service -f
tail -f /home/pi/Hioki_data_logger/watchdog.log

13   Troubleshooting

Status stays red — "No HIOKI device found"

  1. Check that the USB / RS-232 cable is firmly connected at both ends.
  2. Verify the meter is powered on and not in local-only mode.
  3. On Windows, open Device Manager and confirm a COM port appears for the meter.
  4. The app retries automatically every few seconds — wait or reconnect the cable.

Readings show "Timeout Error" repeatedly

The meter is detected but not responding to data queries. Check that the baud rate matches (should be 9600) and that no other program (e.g. HiLogger software) has the COM port open.

Model button shows "Model" (empty)

No product model has been set. Click Model and type the model name, or scan a barcode. If you close this dialog without entering anything, the app will prompt again when the next stable reading is captured.

DB upload keeps failing — "N queued"

The SQL Server is unreachable (network down, server offline, or wrong credentials). Records are queued in pending_uploads.json and will upload automatically once connectivity is restored. CSV data is always saved locally regardless.

Barcode scanner does not change the model

  1. Ensure the scanner is configured for Code 39 Extended with AIM output enabled.
  2. The scanner must output characters quickly (within 100 ms per character) — slow scanners may not trigger decoding.
  3. Check model_changes.csv — if a barcode row appears there the scan was received but the format may be incorrect.

No audio on PASS / FAIL

  1. Confirm ResistancePass_TH.mp3 and ResistanceOver_TH.mp3 are in the app directory.
  2. Check system volume is not muted.
  3. On Raspberry Pi / Linux, ensure GStreamer is installed — run sudo ./setup_services.sh to install it.
  4. Use Ctrl+P or Ctrl+F to test audio without a connected meter.

App does not start / Python error on launch

  1. Confirm all packages are installed: pip install -r requirements.txt
  2. Confirm ODBC Driver 18 for SQL Server is installed (required even if DB is unreachable).
  3. On Raspberry Pi, run the setup script: sudo ./setup_services.sh

Log file grows too large

The on-screen log keeps only the last 500 lines. The CSV files accumulate indefinitely — archive or delete old YYYYMMDD.csv files periodically if disk space is a concern.
